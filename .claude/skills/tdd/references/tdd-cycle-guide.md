# TDDサイクルガイド

TDD（テスト駆動開発）の実践ガイドです。Go言語向けの具体的なパターンを記載します。

## テストリストの書き方

TODOコメントとしてテストファイル（またはその先頭）に列挙します。

```go
package mypackage_test

// テストリスト:
// TODO: 正常系: 有効な入力で正しい結果を返す
// TODO: 正常系: 境界値（最小値）で正しく動作する
// TODO: 異常系: nilが渡された場合エラーを返す
// TODO: 異常系: 空文字列が渡された場合エラーを返す
// TODO: 異常系: 範囲外の値でエラーを返す
```

**ポイント:**
- 完全なリストを作ろうとしない。実装中に気づいたら追加・変更してよい
- ハッピーパス（正常系）から始める
- エッジケースは後から追加する

## テスト選択順序

実装が最もシンプルになるケースから始める。

1. **最もシンプルなハッピーパス** → 仮実装（定数返し）で通せる
2. **別のハッピーパス** → 三角測量で一般化を強制する
3. **エッジケース** → 境界値の処理を追加する
4. **エラーケース** → エラーハンドリングを追加する

## 仮実装から一般化するパターン

### ステップ1: 定数返し（仮実装）

最初のテストを通すだけの最小限のコード。

```go
// テスト
func TestAdd(t *testing.T) {
    result := Add(2, 3)
    if result != 5 {
        t.Errorf("Add(2, 3) = %d, want 5", result)
    }
}

// 仮実装（定数返し）
func Add(a, b int) int {
    return 5  // とりあえずハードコード
}
```

### ステップ2: 三角測量

別のテストケースを追加して一般化を強制する。

```go
// 別のテストケースを追加
func TestAdd_AnotherCase(t *testing.T) {
    result := Add(1, 4)
    if result != 5 {
        t.Errorf("Add(1, 4) = %d, want 5", result)
    }
}
```

両方のテストを通すには一般化するしかない:

```go
// 一般化された実装
func Add(a, b int) int {
    return a + b
}
```

### テーブル駆動テストへの移行タイミング

以下の条件を満たしたらテーブル駆動テストに移行する:
- 同じ関数に対するテストが3件以上ある
- テストの構造が繰り返している（setup → call → assert）

```go
func TestAdd(t *testing.T) {
    tests := []struct {
        name    string
        a, b    int
        want    int
    }{
        {"正の数", 2, 3, 5},
        {"ゼロを含む", 0, 5, 5},
        {"負の数", -1, 1, 0},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got := Add(tt.a, tt.b)
            if got != tt.want {
                t.Errorf("Add(%d, %d) = %d, want %d", tt.a, tt.b, got, tt.want)
            }
        })
    }
}
```

## Refactorチェックリスト（Go向け）

Refactorフェーズで確認する項目。

### 重複の除去
- [ ] 同じコードが複数箇所にある場合、関数として切り出す
- [ ] 定数を使えるマジックナンバーがある場合、定数として定義する

### 命名の改善
- [ ] 変数名・関数名が意図を表しているか
- [ ] Go標準の命名規則に従っているか（`camelCase`、エクスポートは`PascalCase`）

### Go固有の整理
- [ ] エラーハンドリングの整理（`fmt.Errorf` + `%w` でラップ）
- [ ] `defer` の適切な使用（ファイル・DB接続のクローズ）
- [ ] インターフェースを使った依存の抽象化（必要な場合）

### Lint確認

```bash
golangci-lint run
```

よくあるlint指摘:
- `errcheck`: エラーを無視している（`_ = func()` パターン）
- `unused`: 使われていない変数・関数
- `govet`: 疑わしい構造体タグ等

## `go test` コマンド一覧

```bash
# 現在のパッケージのテストを実行
go test .

# サブパッケージを含む全テストを実行
go test ./...

# 詳細出力（テスト名・PASS/FAILを表示）
go test -v ./...

# 特定のテストのみ実行
go test -run TestFunctionName ./...

# レースコンディション検出（最終確認時）
go test -race ./...

# カバレッジ計測
go test -cover ./...

# カバレッジのHTML出力
go test -coverprofile=coverage.out ./...
go tool cover -html=coverage.out

# キャッシュを無効にして再実行
go test -count=1 ./...
```

## よくある間違いと対策

### 間違い: テストが通る前にリファクタリングを始める

**症状**: Greenになる前にコードを整理しようとして、どこで失敗しているか分からなくなる。

**対策**: 必ずGreen（テストPASS）を確認してからRefactorを始める。

### 間違い: 複数のテストを同時にRedにする

**症状**: 複数の失敗を同時に修正しようとして、どこを直せばいいか分からなくなる。

**対策**: テストリストから1件だけ選び、それだけをRedにする。

### 間違い: 「完璧な」実装を最初から書こうとする

**症状**: 最初から一般的な実装を書こうとして、テストが複雑になる。

**対策**: 仮実装（定数返し）から始めて、三角測量で一般化する。

### 間違い: Refactor中にテストを変更する

**症状**: プロダクションコードと一緒にテストも書き直して、テストの意味が変わってしまう。

**対策**: Refactorフェーズでは、プロダクションコードのみを整理する。テストを変更したい場合は、一度Red-Greenサイクルを完了してから別途対応する。
