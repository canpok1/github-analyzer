# TDD スキル

## 役割

テスト駆動開発（TDD）のサイクルに従い、実装タスクを進めます。
ベビーステップで確実に動作するコードを積み上げます。

## ワークフロー

### Phase 1: テストリスト作成

実装前に、必要なテストケースをTODOコメントとして列挙します。**この段階ではコードを書かない。**

```go
// TODO: 正常系: 正しい入力で期待値を返す
// TODO: 異常系: nilが渡された場合エラーを返す
// TODO: 境界値: 空文字列の場合の動作
```

- シンプルなケースから複雑なケースの順に並べる
- テストリストは途中で追加・変更してよい
- 全テストケースを実装する前に開始する

### Phase 2: Red-Green-Refactor サイクル

テストリストから1件ずつ取り上げ、以下のサイクルを繰り返す。

#### Red: 失敗するテストを書く

1. テストリストから最もシンプルなケースを1件選ぶ
2. テストコードを書く
3. `go test ./...` を実行して**失敗**を確認する
4. コンパイルエラーもRedとみなしてよい

**制約**: 一度に複数のテストをRedにしない

#### Green: 最小限のコードで通す

1. テストが通る最小限のプロダクションコードを書く
2. 仮実装（定数返し）でもよい
3. `go test ./...` を実行して**PASS**を確認する

**制約**: テストを書く前にプロダクションコードを書かない

#### Refactor: コードを整理する

1. テストが通ったまま、コードの重複を除去・整理する
2. `go test ./...` でテストがPASSしたままであることを確認する
3. `golangci-lint run` でlintエラーがないことを確認する

**制約**: Refactor中はテストコードを変更しない

#### コミット前チェック

Green または Refactor 後、コミットする前に以下を実行する：

1. `gofmt -l .` でフォーマットチェック → 出力があれば `gofmt -w .` で修正
2. `golangci-lint run` でlintチェック → 指摘があれば修正
3. 修正した場合は `go test ./...` で既存テストが壊れていないことを確認

#### コミット

- コミット前チェックが通った後にコミットする
- **Red状態ではコミットしない**
- コミットメッセージにはIssue番号を含める

### Phase 3: 完了確認

全てのテストケースが実装済みになったら最終確認を行う。

```bash
# 全テストを実行（レースコンディション検出）
go test -v -race ./...

# lintチェック
golangci-lint run
```

## 制約まとめ

- テストを書く前にプロダクションコードを書かない
- 一度に複数のテストをRedにしない
- Red状態でのコミット禁止
- Refactor中はテストを変更しない
- Greenにするために定数を返す仮実装はOK
- コンパイルエラーのままRedとみなすのはOK
- テストリストを途中で追加・変更するのはOK
